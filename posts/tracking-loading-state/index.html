Tracking Loading State

September 21, 2020

Problem: when developing SPAs we need a way to manage UX paths dependent on asynchronous request states.

Basic solutions

dataset.length

<template>
	<main>
		<registration-component v-if=isNotLoading />
		<loading-animation v-if=isLoading />
	</main>
</template>
<script>
export default {
	// ...,
	computed: {
		isLoading() {
			return this.data.length == 0;
		}
		// ...
	}
}
</script>

This is dirt simple to implement and understand. It also avoids some of the technical and cognitive overhead of implementing a programmatic state management or message polling system.

Some obvious shortsides pop up if you need to track the loading state of more than one variable: loading what? If (and only if) everything can and should be hidden from the user prior to loading, the state could be managed under a singular loading variable, but this can falter in the face of the postback.

It provides little mechanism for request error handling, but for the most part I’m leaving error handling out of this post because it’s an entirely different state management problem. A harder one ðŸ˜¨

This addresses the issues of emergent loading states, but depending on how much the page is loading this method becomes cumbersome without a programattically-derived state management system, eg:

let loadingFields 
	= [ ['academicYears', api.getAcademicYears]
		, ['states', api.getStates]
		, ['accountInfo', api.getAccountInfo]
		];
 
state = { 
	...loadingFields.reduce((acc. val) => ({...acc, val: null}))
}
// more reducers for mutations, computed getters, etc

But programmatically deriving states creates an even less deterministic (that is, less easily reasonable) page state. Being unable to reason about these state transitions is guaranteed to lead to bugs. How many states are in the example above this, with loadingStates and loadingGenders? I count four base cases:

  • isLoadingStates = false; isLoadingGenders = false
  • isLoadingStates = true; isLoadingGenders = false
  • isLoadingStates = false; isLoadingGenders = true
  • isLoadingStates = true; isLoadingGenders = true;
  • isLoadingStates = true (error) …

Once the error state is considered the state permutations explodes. Not by a ton in this, going from 4â†’9, but again this error state is untracked here, so debugging an error arising from nine possible states (out of four explicitly defined) takes a heckuva lotta work, perhaps moreso than if the states had been modeled to begin with. But that’s 9 states to model! For a two element form in this case!

Again, not getting into error management, but I think a good request management paradigm will account for errors, so ideally there is a place to send them. One system I’ve experimented with, that can be spread-merged into an existing KV-based store like Vuex’s, or merged in via mixins to a Vue component goes something like this:

const DATA_REQUESTED = Symbol('DATA_REQUESTED');
const DATA_RECEIVED  = Symbol('DATA_RECEIVED');
const ERROR_RECEIVED = Symbol('ERROR_RECEIVED');

const state = { history: [] };

const mutations = { 
	requestData: (state, data) => state.history.unshift(
		{event: DATA_REQUESTED, data}
	)
	, recordData: (state, data) => state.history.unshift(
		{event: DATA_RECEIVED, data}
	)
	, recordError: (state, data) => state.history.unshift(
		{event: ERROR_RECEIVED, data}
	)
};

const getters = {
	receivedData: state => state.history[0].event == DATA_RECEIVED
	, loadingData: state => state.history[0].event == DATA_REQUESTED
	, receivedError: state => state.history[0].event == ERROR_RECEIVED 
};

This is the basic outline. It uses a stack-like array to organize history in descending order so time traveling I mean auditing or request-state error logging can be implemented. It might be used then something like this:

methods: {
	getStates() {
		this.requestData(api.statesUrl);
		fetch(api.statesUrl)
		.then(response => {
			if(!response.ok) {
				this.recordError(`Error fetching states: ${response.status}`);
			} else {
				const data = response.json();
				this.recordData(data);
			}
		});
	}
}
, mounted() {
	this.getStates();
	this.getGenders();
}

Atomic solutions

<template>
	<div>
		<select v-if=notLoading>
			<option v-for="state in states">{{ state }}</option>
		</select>
	</div>
</template>
<script>
export default {
	name: 'state-selector'
	// ...
}
</script>

This example is the same as the others but loading state is handled by the component that actually needs the data: abstract the select box into its own component so that it manages its own readystate. This definitely cleans up the page level code a lot, but for derivative states it would require a finite state machine or event bus system to compute/signal the sorts of page-level states that would dis/enable the submit button. It’s not hard to plug this into a vuex store, managing a single isLoadingStates prop.

“WAIT ISN’T THAT THE SAME AS ABOVE”

Sort of. It separates concerns such that the page doesn’t care if there is states data, leaving that to the concern of the states component, but can still determine if the page itself is in a ready state. If all the components are ready to go we only need to know if the form is valid and then submission can be enabled.

This also allows components to easily come up with fallbacks. A question begged by many of these approaches is that the user shouldn’t be interacting with pages that aren’t completely loaded, but a user (more specifically, a user’s tools) just wants to get on with things and often doesn’t need the helper data we are loading. They’re often automating form completion with browser/credential managers' form-filling functionality. More sophisticated users (a growing population as Zoomers come of age) are even automating their bureaucratic chores with scripts and bots.

With this in mind, we should have a text field for state and swap it out for the dropdown when the states are loaded, matching to any typed state data:

<template>
	<div>
		<select v-if=notLoading v-model=selectedState>
			<!-- ... -->
		</select>
		<input v-if=loading type=text v-model=typedState />
	</div>
</template>

Component wrapper

<template>
	<div>
		<component 
			v-if=notLoading
			:is=component 
			:data=loadedData 
			v-on=$listeners 
		/>
		<loading-animation v-if=loading />
	</div>
</template>
<script>
export default {
	name: 'data-loading-component'
	, props: ['component', 'generator']
	, data() {
		return { 
			loading: true
			, loadedData: null
		};
	}
	, created() {
		generator().then(
			d => {
				this.loadedData = d;
			}
		);
	}
}
</script>

This is a data-loading component. Written as is, it takes a component name and a promise-returning data-resolving function as props and manages the loading state for the internal component. When the promise resolves, it feeds the data to the component and renders it. Another option is to use a conditionally rendered default slot instead of a component element.

Using vm.$listeners allows for messages to be passed up the chain, for example from a validated input or some such.

This model can also be adapted to using a store or event bus or other system for communicating updated states and data to children, and it allows for derivative states using a method similar to the validation-observer component from vee-validate consume all states as one or individually.

Here are the two main examples I’ve outlined of its use, first as defined above and second with slots:

<template>
	<form>
		<data-loaded-component-with-props
			:generator="api.getStates"
			component="state-selector"
		/>
		<data-loaded-component-with-slots
			:generator="api.getStates"
		>
			<select name=gender slot-scope=genders>
				<option v-for="gender in genders">{{ gender }}</option>
			</select>
		</data-loaded-component>
	</form>
</template>

Other solutions

FSM-driven

The xstate library produces another way to approach this. I don’t want to make this article too specialized by focusing on any singular library (even though much of this has been directed at Vue development), but its event-driven state charts can provide a unique solution, especially in multi-part SPAs, where advancing to a “stage” of the application is user and/or network driven. It’s design choice to make promises, async generators and Rx observables first-class state machines that can compose with more traditional FSM paradigms really makes it shine here.

The Reddit API tutorial is a great place to see examples of the paradigm.

Workers

An idea I’ve only just begun to toy with is sourcing all frontend API requests through a worker. On page initialization the worker is initialized and maintains an internal state of requests and their states. This is the approach stockroom uses. Workers have the very clear benefit of removing state and request computation from the UI thread, making SPAs more performative over all, but they also look a bit like magic in a repo, perhaps owing to their underutilization.

There are examples out there, including one I found specifically targeting concurrent vuex architecture, of how to deploy state machines. With the web worker’s limitated API, message-based FSM machines seem like a natural venue for exploration.

Conclusion

None. This is an ongoing discussion and the ideal solution can only be identified on a problem-by-problem basis. However, I hope this has provided some value in the form of perhaps illuminating some previously unconsidered solutions or expanding on their limitations and implicit expectations.
